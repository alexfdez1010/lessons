## Section 1 — Why Bitcoin Leaks

Bitcoin is where the whole story starts: a transparent, append-only ledger that hides nothing and forgets nothing. If you can't say *exactly* why pseudonymity isn't privacy, nothing downstream makes sense.

## Section 3 — The Cryptography

Zero-knowledge proofs are the engine. The three properties are not interchangeable, and confusing soundness with completeness is the single most common slip — so this section hunts for exactly that.

<MatchConcepts
  client:visible

    { term: 'Soundness', definition: 'A dishonest prover cannot prove a FALSE statement, except with negligible probability.' },
    { term: 'Zero-knowledge', definition: 'The verifier learns nothing beyond the truth of the statement — never the witness.' },
    { term: 'Witness', definition: 'The private secret the prover holds — a key, password, or solution — never revealed.' },

  ]}
  explanation="Completeness = honest claims go through. Soundness = liars cannot bluff. Zero-knowledge = nothing leaks beyond truth. The witness is the secret; succinct is about proof size and verification speed, not hiding data."

## Section 4 — Inside the Machine

Notes, commitments, nullifiers, the Merkle tree, value commitments, the pools. This section refuses to let the commitment and the nullifier be swappable — because they do opposite jobs.

<MatchConcepts
  client:visible
  question="Match each cryptographic object to its exact role. The commitment and the nullifier are NOT interchangeable."
  pairs={[
    { term: 'Note', definition: 'The shielded analog of a UTXO — sealed value belonging to a z-address.' },
    { term: 'Note commitment', definition: 'A public, hiding-and-binding fingerprint that proves a note exists without revealing its value.' },
    { term: 'Nullifier', definition: 'A unique per-note "spent" marker that prevents double-spends without revealing which note it came from.' },
    { term: 'Merkle tree', definition: 'The append-only structure a spend proves membership in, without disclosing which leaf.' },
    { term: 'Value commitment', definition: 'A homomorphic seal on an amount; inputs and outputs must sum to zero to prove balance.' },
  ]}
  explanation="Commitment = the note's birth fingerprint (hiding + binding). Nullifier = the note's one-time death stub (anti-double-spend, unlinkable). Swap them and the whole privacy/integrity story collapses."
/>

<MCQ
  client:visible
  question="On-chain, a z→z spend reveals a nullifier. Why does publishing it NOT let an observer link the spend back to the note that was created earlier?"
  options={[
    { text: 'The nullifier is derived so it cannot be tied to the note commitment it corresponds to', correct: true },
    { text: 'The nullifier is encrypted, and only full nodes hold the key to read it' },
    { text: 'The nullifier reveals the note but the network deletes the link after one block' },
    { text: 'The nullifier is the same value as the commitment, so there is nothing new to link' },
  ]}
  explanation="There's exactly one valid nullifier per note, and nodes reject a repeat (double-spend). But it deliberately does NOT reveal which note it came from — that's how you get integrity without visibility. It is emphatically not the commitment."
/>

A precision trap that makes cryptographers wince when stated wrong.

<MCQ
